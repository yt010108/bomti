Exact score contract (`bomti_index_v1`):

- Luna returns five 0-100 risk dimensions: `contextMismatch` 25%, `genericityCliche` 25%, `credibilityRisk` 20%, `specificityGap` 20%, `toneReadabilityRisk` 10%, plus segment-ID evidence and critical flags.
- Terra returns a 0-100 holistic Bomti index, segment-ID evidence, explanation, and the same critical-flag vocabulary.
- DeepSeek returns the same five dimensions and holistic index in one candidate. The server normalizes it to `bomti_index_v1`, then projects only index, descriptor, concise explanation, at most three evidence findings, and improvement directions to the guest response.
- Without escalation, final index is `round((dimensionAggregate + holisticIndex) / 2)`.
- Sol is required when the two inputs differ by at least 15 points or disagree on `fabrication_or_unverifiable_claim`; Sol returns the final normalized contract. If Sol is unavailable or capped, mark the evaluation `failed_needs_adjudication`, refund the reserved paid allowance, and show an explicit retry-later state.
- Descriptors are `0-24 밤티 거의 없음`, `25-49 살짝 밤티`, `50-74 꽤 밤티`, `75-100 밤티 그 자체`; descriptors are explanatory, never pass/fail.
- Critical flags are exactly `fabrication_or_unverifiable_claim`, `context_mismatch`, `cliche_saturation`, `unsupported_superlative`, `privacy_leak`, and `prompt_injection_attempt`; only the first participates in automatic Sol escalation in v1.
- Before pseudonymization, normalize input to NFC and split the answer into immutable sentence records `{segmentId, originalText}`. Send `{segmentId, pseudonymizedText}` to models and require evidence to cite `segmentId`, never byte/string offsets. Immediate results may highlight `originalText` held only in browser memory; persisted history stores only pseudonymized segments and segment-ID evidence.

The server computes descriptor from `finalIndex`; Sol may alter only paths listed in its decisions.
- `FinalProvenancePath` is exactly `/finalIndex`, `/descriptor`, each of the five literal `/dimensions/<DimensionName>` paths, each of the five literal `/dimensionExplanations/<DimensionName>` paths, `/explanation`, `/evidence`, `/improvements`, `/fragments`, and `/criticalFlags`. `ProvenanceSource = 'luna'|'terra'|'sol'|'server:hybrid'|'server:range'|'server:union'`. `NormalizedVerdict = {contractVersion:'bomti_index_v1',finalIndex:integer[0..100],descriptor:'밤티 거의 없음'|'살짝 밤티'|'꽤 밤티'|'밤티 그 자체',dimensions:Record<every DimensionName,integer[0..100]>,dimensionExplanations:Record<every DimensionName,string[1..400]>,explanation:string[1..800],evidence:Evidence[0..5],improvements:Improvement[1..5],fragments:Fragment[0..2],criticalFlags:CriticalFlag[0..6],provenance:Record<every FinalProvenancePath,ProvenanceSource>}`. Every `FinalProvenancePath` appears exactly once. A Sol decision for one `/dimensions/<DimensionName>` changes only that child path to `sol`; sibling dimension paths retain their source, so mixed per-dimension provenance is representable. Sol cannot decide `/descriptor` or `/dimensionExplanations/*`; those remain server range or Luna-derived respectively.
- Without Sol, dimensions/explanations and one improvement per dimension come from Luna; holistic explanation/fragments and distinct additional improvements come from Terra; final index is the hybrid average; descriptor is server range mapping; critical flags are the declared-enum-order union. Evidence deduplicates by NFC `(segmentId,dimension,summary)`, sorts severity descending then segment ordinal then dimension enum, and takes five. Improvements deduplicate by NFC `(dimension,direction)`, sort by dimension enum with Luna before Terra, and take five. With Sol, start from this baseline and replace only roots named by valid `decisions`; altered roots are `sol`, untouched roots retain provenance, descriptor stays `server:range`, and missing/duplicate/contradictory decisions reject the candidate.
- Unicode limits count normalized Unicode code points, not UTF-16 units or bytes. Sentence segmentation is deterministic: NFC-normalize, preserve paragraph order, split on Korean/English sentence terminators followed by whitespace/end plus paragraph boundaries, retain punctuation, assign `s0001...`; empty fragments are discarded and unsplittable text becomes one segment.
- DeepSeek dimension aggregate uses the same fixed weights and its holistic index uses the returned validated value; final guest index is the same rounded average formula. Sol replaces, rather than averages with, the disputed final fields; undisputed evidence is retained only when its segment ID remains valid, with provenance recorded.
- Provider limits are guest 12,000 input/1,500 output tokens/15s/no reasoning, Luna 24,000/1,800/30s/medium, Terra 24,000/1,800/30s/high, and Sol 16,000/2,200/45s/high. Before dispatch the server creates an opaque UUID `clientCorrelationId`, persists it in encrypted form, and sends it in provider metadata/idempotency fields when supported. Retry once only when the transport proves the request was not accepted. Timeout or disconnect after possible acceptance is never retried. If no provider request ID was received, persist a nullable request ID plus the encrypted client correlation key in `unresolved_reserved`, retain the full budget reservation, return a non-success terminal state, and prevent duplicate calls. Reconciliation first queries by provider request ID, otherwise by client correlation where supported; if neither provider lookup is possible, an operator must choose `accepted` with verified token/cost evidence or `rejected` with verified provider absence through the server-only reconciliation command. Until that explicit resolution the reserve remains held and alerts repeat; there is no automatic release.
- Stable terminal mapping is `completed`, `validation_failed`, `consent_required`, `quota_exhausted`, `budget_disabled`, `provider_unavailable`, `provider_output_invalid`, `cancelled_before_acceptance`, `failed_refunded`, and `failed_needs_adjudication`; only `completed` returns a verdict.
- HTTP/error mapping is `completed→200`, `in_flight_before_acceptance→202 EVALUATION_IN_PROGRESS`, `validation_failed→400 INPUT_INVALID`, `consent_required→428 CONSENT_REQUIRED`, `quota_exhausted→429 GUEST_LIMIT|ACCOUNT_LIMIT|GLOBAL_LIMIT|SOL_LIMIT`, `budget_disabled→503 PAID_EVALUATION_DISABLED`, `provider_unavailable→503 GUEST_PROVIDER_UNAVAILABLE|AUTH_PROVIDER_UNAVAILABLE`, `provider_output_invalid→502 PROVIDER_OUTPUT_INVALID`, `cancelled_before_acceptance→499 REQUEST_CANCELLED` in server state and an aborted client response, `failed_refunded→503 EVALUATION_FAILED_REFUNDED`, and `failed_needs_adjudication→503 ADJUDICATION_REQUIRED`. A duplicate guest POST with the same idempotency key before acceptance returns `202` and `{status:'in_flight',code:'EVALUATION_IN_PROGRESS',retryAfterMs:1000}` with `Retry-After: 1`; the client may repeat the identical POST/key until it receives a terminal response. No status URL, request body, or verdict is persisted. At or after first possible acceptance, every duplicate/concurrent guest POST returns `409` and `{status:'terminal',code:'GUEST_ATTEMPT_ALREADY_USED'}` with no prior verdict.
- Feedback reason codes are exactly `clear_explanation`, `useful_evidence`, `actionable_improvement`, `score_felt_wrong`, `evidence_felt_wrong`, `not_actionable`; benchmark rating rationale codes are exactly `context_fit`, `specificity`, `credibility`, `cliche`, `tone_readability`, `other_reviewed`. Free-text feedback/rationales are forbidden in v1.

Quota, refund, and cost matrix:

| Terminal path | IP/cookie/global guest | Account | Sol | Budget |
| --- | --- | --- | --- | --- |
| completed guest | consume all once | n/a | n/a | accepted guest cost retained |
| completed auth without Sol | n/a | consume once | n/a | accepted Luna/Terra cost retained |
| completed auth with Sol | n/a | consume once | consume after acceptance | all accepted cost retained |
| validation/consent/quota/budget rejection or cancel before acceptance | no consumption | no consumption | no consumption | release all reservations |
| provider rejection proven before acceptance | refund guest/account/global | refund account | release | release unaccepted reservation |
| ambiguous acceptance, invalid output, timeout, or cancel after acceptance | refund user-facing guest/account allowance; keep anti-abuse attempt marker for the bucket | refund account allowance | consume only if Sol acceptance is known; otherwise hold until reconciliation TTL | retain worst-case reservation until reconciliation, then settle accepted cost or release |
| Sol required but capped/unavailable | n/a | refund account allowance | no consumption | retain accepted Luna/Terra cost; release Sol reserve |
| expired reservation | release allowance unless an accepted request is later reconciled; duplicate idempotency remains blocked | same | same | settle known accepted cost, otherwise release |

- IP, cookie, global, account, Sol, and budget reservations are atomic. Duplicate idempotency keys never create another call or consumption. Guest/global and account display buckets use KST as configured; the provider budget uses a UTC calendar month. Before any call, reserve the worst-case cost from configured input/output ceilings for every call that may occur, including Sol; release the difference only after settlement. Pricing variables and `PROVIDER_PRICING_VERSION` are mandatory when paid inference is enabled.
- Guest duplicate idempotency never returns a stored verdict: before provider acceptance the duplicate POST follows the exact `202 EVALUATION_IN_PROGRESS` retry contract above; at/after possible acceptance or terminal completion it receives `409 GUEST_ATTEMPT_ALREADY_USED`. Accepted or ambiguously accepted guest attempts keep only hashed anti-abuse/idempotency proof for seven days.
- Cost is integer microdollars using ceiling division for each input/output component; no floating point. The reservation belongs to the UTC month in which the first provider request is accepted and never moves at month rollover. Luna accepted/Terra rejected retains Luna cost and refunds account allowance; both primary calls accepted/Sol unavailable retains primary cost and refunds account allowance; any accepted Sol retains Sol allowance/cost even if output is invalid. Repeated ambiguous failures never retry or create a new reservation. Unknown acceptance keeps its full reserve beyond 600 seconds in `unresolved_reserved` and blocks that budget until retrieval or operator resolution, never auto-releases. Late reconciliation may only settle cost/release reserve and delete the reconciliation row; it never creates a verdict or re-consumes a refunded user allowance.

Benchmark and calibration contract:

- Version `bomti_benchmark_v1` defines record `{recordId,groupId,questionClass,targetRoleClass,answerSegments,verdict,anonymizationVersion,provenanceClass,reviewStatus}`, pair `{pairId,leftRecordId,rightRecordId,groupId,systemChoice:'left'|'right'|'tie'}`, rating `{pairId,raterAlias:/^r[0-9]{3}$/,choice:'left'|'right'|'tie'|'abstain',rationaleCodes[]}`, and usefulness `{rating:1..5,reasonCode}`. Random `groupId` is assigned only after irreversible anonymization by the offline curation step; no hash of account/history context, evaluation ID, precise timestamp, or rater identity mapping appears in benchmark storage/export.
- Single contributed evaluations enter only the eligible record pool. `benchmark:pair` may propose pairs only within an operator-assigned anonymous `groupId` and matching generalized question/role classes; an operator must curate/import them. `systemChoice` is `tie` for an inclusive absolute index difference of 0-5, otherwise `left` or `right` for the higher Bomti index.
- Per pair, human choice is the strict majority of non-abstaining ratings; tied votes/all-abstain are missing. Every metric emits `{numerator,denominator,missing}` and a null rate when denominator is zero. Pairwise agreement uses matching non-missing human/system choices over curated pairs; missing is curated pairs with tied/all-abstain human choice or absent system choice. Evaluator disagreement uses non-unanimous pairs over pairs having at least two non-abstaining ratings; missing is curated pairs with fewer than two. Descriptor distribution uses completed verdicts in each range over all completed verdicts; missing is eligible evaluation attempts without a completed verdict. Escalation uses auth attempts that invoked Sol over auth attempts with valid Luna+Terra; missing is auth attempts lacking either valid primary candidate. Failure-category uses each terminal non-completed category only among requests that passed validation and consent over all requests that passed validation and consent; validation/consent failures are reported separately as pre-eligibility counts and never enter its numerator. Missing is passed-validation/consent requests having no terminal state. Usefulness uses ratings 4-5 over submitted ratings; missing is completed auth evaluations without feedback. Local corpus is three synthetic pairs; real pairs remain `operator_not_supplied`.

Authentication contract:

- Google scopes are exactly `openid email profile`. Allowed redirect origins are exact members of `AUTH_REDIRECT_ALLOWLIST` plus `APP_ORIGIN`; wildcard origins are forbidden. Callback errors are `AUTH_STATE_INVALID`, `AUTH_CODE_EXCHANGE_FAILED`, `AUTH_EMAIL_MISSING`, `AUTH_SESSION_EXPIRED`, and `AUTH_REDIRECT_DENIED`.
- Google access/refresh tokens are never persisted or logged. Supabase session cookies are server-set `HttpOnly`, `SameSite=Lax`, `Secure` outside local test, `Path=/`, with expiry matching the Supabase session; middleware refreshes expiring sessions and rotates cookies.
- PKCE state and code verifier live only in encrypted, `HttpOnly`, `SameSite=Lax`, 10-minute cookies; callback compares state in constant time and deletes both cookies after one use. Browser code never receives refresh/provider tokens; server components, route handlers, and middleware alone call `@supabase/ssr` and rotate session cookies.
- Account deletion requires `auth_time` no older than 10 minutes. Otherwise start Google reauthentication with `prompt=login&max_age=0`. The idempotent saga is `requested → sessions_revoked → app_data_deleted → auth_user_deleted → complete`: create a server-only encrypted deletion job, deny all requests for that subject including stale JWTs, revoke refresh sessions, aggregate accepted cost, delete linkable app rows, call the Supabase admin deletion API, and retry with bounded backoff if admin deletion fails. The transition to `auth_user_deleted` atomically nulls `encrypted_auth_user_id`; afterward the row contains only the one-way subject block marker, retry metadata, and TTL. That marker survives only until the maximum access-token TTL, then the whole job self-deletes. Tests inject failure after every transition, verify ciphertext is absent after auth deletion, and prove eventual removal without restoring data.
